GCC 优化级别的核心行为

  • -O1（基础优化）

    • 行为：试图在不消耗过多编译时间的前提下，同时优化执行速度和代码体积。

    • 核心手段：死代码消除、基础分支优化、常量折叠。

    • 体积影响：相比于不优化（-O0），生成的二进制文件体积通常会显著缩小。

  • -O2（标准优化）

    • 行为：生产环境的标准选择。开启所有 -O1 的优化，外加几乎所有不涉及“空间换时间”（即不造成代码显著膨胀）的性能优化。

    • 核心手段：指令调度、严格别名分析（Strict Aliasing优化）、公共子表达式消除。

    • 体积影响：代码体积通常与 -O1 相当，可能略大或略小，整体保持紧凑。

  • -O3（激进优化）

    • 行为：绝对的速度导向。在 -O2 基础上，强制开启为了压榨执行速度而牺牲代码体积的激进优化选项。

    • 核心手段：激进的函数内联、循环展开、自动向量化。

    • 体积影响：通常会导致二进制文件体积显著膨胀（Code Bloat）。

优化手段为何会影响代码体积？

编译器影响代码体积的逻辑分为两个截然相反的方向：“剔除冗余”使其变小，以及“空间换时间”使其变大。

1. 导致代码体积缩小的机制（主要体现在 -O0 切换至 -O1 / -O2）

  • 死代码消除（Dead Code Elimination）：

    编译器通过静态数据流分析，直接丢弃永远无法执行到的分支（如计算结果恒为假的条件判断）和从未被读取的变量定义，从而物理上减少了生成的汇编指令条数。

  • 高效的寄存器分配（Register Allocation）：

    -O0 级别下，编译器会将局部变量频繁地在栈内存与 CPU 寄存器之间来回搬运（大量的 Load/Store 指令）。优化后，编译器会通过图着色等算法，让数据尽可能长期驻留在寄存器中，这直接消除了海量的内存存取指令。

  • 常量折叠与传播（Constant Folding & Propagation）：

    在编译期预先计算出常量表达式的值。例如将汇编级别的多次数学运算指令，直接替换为一条加载立即数的单指令。

2. 导致代码体积膨胀的机制（主要体现在切换至 -O3 时）

  • 函数内联（Function Inlining）：

    为了彻底消除函数调用的上下文切换开销（压栈、保存寄存器、分支跳转、出栈），编译器会将目标函数的完整指令序列直接“复制粘贴”到每一个调用该函数的地方。如果一个 50 行汇编的函数被调用了 100 次，内联后就会凭空产生 5000 行汇编代码。

  • 循环展开（Loop Unrolling）：

    为了减少循环控制变量的增量计算、比较判断，并降低分支预测失败带来的流水线停顿，编译器会将循环体在物理上复制多份。例如，将单次循环执行 100 次，转换为物理代码连写 4 次，执行 25 轮。这直接导致该段指令体积翻倍。

  • 自动向量化（Auto-Vectorization）：

    将标量循环转换为 SIMD（如 ARM NEON 或 x86 AVX）指令。为了确保向量化指令能够正确处理数据内存未对齐的情况，以及处理无法被向量长度整除的”余数”循环（Loop Peeling/Tailing），编译器会生成大量的 Setup、边界检查和 Cleanup 前序/后序代码。核心逻辑变快了，但周围的基础设施代码大幅增加。

实验验证

以下实验使用 GCC 11.4.0（Ubuntu 22.04），在 x86-64 平台上对 6 个测试程序分别以 -O0 / -O1 / -O2 / -O3 编译，测量二进制文件大小、.text 段大小和执行时间（取 3 次运行的最优值）。

测试程序 (bench_overall.c) 包含矩阵乘法 (256×256)、快速排序 (104 万个元素)、字符串处理和控制流密集型递归 (fib(40))。

六、常量折叠与传播（Constant Folding & Propagation）实验

测试程序 (demo_constfold.c) 对比两个函数的 2 亿次调用：

  • const_heavy(x)：参数含宏常量 HEAVY_EXPR，编译器可完全在编译期求值
  • non_const(x, y)：同样逻辑，但第二个参数为运行时变量

执行时间

优化级别耗时加速比 (vs -O0)
-O01.250 s1.00x
-O10.240 s5.21x
-O20.190 s6.58x
-O30.190 s6.58x

分析与机制

HEAVY_EXPR = ((1024*512/32) + (4096-2048)) * 3 + (1<<10) - 999 = ((16384) + (2048)) * 3 + 1024 - 999 = 18432 * 3 + 25 = 55321。

在 -O1 及以上，编译器在编译期计算出 HEAVY_EXPR = 55321，将 const_heavy(x) 变为 return x + 55321，省去了 6 次算术运算。而 non_const 的 y 是运行时参数，编译器无法做等效折叠。

关键发现

  1. 性价比最优窗口是 -O2：在多数场景中，-O2 已获得了绝大部分性能提升（通常 >90% 的 -O3 性能），而代码体积与 -O0 基本持平或更小。

  2. -O1 的效果被严重低估：死代码消除 + 寄存器分配 + 常量折叠在 -O1 上就已启用，单个优化可带来 5x~25x 的加速。对于 I/O 密集型或无内联需求的程序，-O1 是非常实用的选择。

  3. -O3 的边际收益递减：虽然 -O3 能通过循环展开和向量化进一步提升数值计算密集型代码的性能（约 15%~20%），但 .text 段膨胀通常也在 20% 以上。是否启用 -O3 取决于 target 的缓存大小和对代码体积的敏感程度。

  4. 内联是双刃剑，但有联动收益：函数内联虽然消除栈帧开销是最大收益来源，但更关键的是内联后为编译器提供了更大的优化窗口——展开后的代码可以进一步接受常量折叠和公共子表达式消除，从而生成比原始函数调用序列更紧凑的代码。
