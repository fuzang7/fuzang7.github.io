结构思维而非流程思维

2026年1月17日6分钟阅读

  • 日志

训练“结构直觉”本质上是训练一种**“降维打击”的能力：从繁杂的过程（Process）中抽离出静态的状态（State）**。

大多数人的大脑是“叙事型”的（先做A，再做B，然后C），而架构师的大脑是“地图型”的（A、B、C 在空间上是如何布局和连接的）。

以下是 4 个通用的、可落地的训练方法，从易到难，强制你切换大脑模式：

方法一：“切断电源”思维训练 (The Power-Off Test)

这是最直观的训练法。在设计任何系统（无论是代码、写书还是管理团队）时，想象下一秒突然断电。

  • 核心问题： 为了在通电后能完美恢复刚才的一切，我必须在“断电前”把什么东西存下来？

  • 训练逻辑：

    • 如果是“功能思维”，你会想“我还没运行完那个循环”。（这是错的，进程死了循环就没了）。

    • 如果是“结构思维”，你会想“我需要记录当前处理到了第几行（Cursor），以及当前的累加值（Accumulator）”。

  • 通用案例：

    • 写小说： 不要只想“主角要去杀恶龙”（剧情）。要想“如果我现在停笔，读者脑子里必须留存哪些关于主角的信息（血量、装备、心理阴影），下一章才能接得上？” —— 这些信息就是你的人物卡（Struct）。

    • 做业务： 不要只想“我要把货卖出去”（流程）。要想“如果销售离职了，为了让新人无缝接手，CRM 系统里必须存留什么字段？” —— 这就是数据库设计。

方法二：“名次优于动词”约束法 (Nouns Over Verbs)

在设计阶段，给自己设定一个强制约束：禁止使用动词。

  • 原理： 动词（计算、发送、显示）代表逻辑，是易变的；名词（订单、用户、哈希表）代表实体，是稳定的。结构由名词决定。

  • 训练步骤：

    1. 拿出一张白纸。

    2. 把你想要的功能描述出来（例：“用户登录后可以发布文章”）。

    3. 划掉所有动词（登录、发布）。

    4. 只剩下名词（用户、文章）。

    5. 连线： 用户和文章是什么关系？（一对多）。文章里包含什么属性？（标题、内容、时间戳）。

    6. 最后才加动词： 动词只是在这些名词之间搬运数据的“搬运工”。

方法三：阅读顶级项目的“骨架” (Skeleton Reading)

不要读逻辑代码（.c/.cpp/.py），只读头文件（.h）或数据定义文件（.schema / .proto）。

  • 操作指南：

    • 去 GitHub 下载 Redis 的源码。

    • 绝对不要看它是怎么处理命令的。

    • 直接打开 server.h (或旧版的 redis.h)。

    • 你会看到 redisServer 和 redisClient 这两个巨大的结构体。

    • 思考： 为什么它要把 dict *commands 放在这里？为什么 client 结构体里有一个 list *reply？

  • 顿悟时刻： 当你看着这些结构体，脑子里能自动“脑补”出整个 Redis 是怎么运转的时候，你的结构直觉就练成了。

方法四：输入/输出黑盒法 (The Black Box Transformation)

把你的系统看作一个纯粹的数学函数：f(Datain​)=Dataout​。

  • 训练逻辑： 忽略中间的所有步骤。

    • 输入是什么？ （例如：一串乱序的 HTTP 请求字节流）。

    • 输出是什么？ （例如：一个结构化的 JSON 对象）。

    • 中间状态是什么？ 为了把那个乱序的流变成 JSON，中间需要什么容器来暂存？（这就推导出了 Buffer、Parser State 等结构）。

  • 价值： 这能帮你过滤掉“界面”、“交互”等干扰项，直击数据流动的本质。

举个通用的例子：设计一个“电梯系统”

  • 普通人（功能思维）：

    • 有人按按钮 → 电梯动 → 到了开门 → 关门。

    • 代码写出来就是一堆 if (button_pressed) move_up() 的面条代码。

  • 架构师（结构思维 - 使用上述方法）：

    • 切断电源法： 如果电梯停电了，来电后它怎么知道该去哪？

    • 名词约束法： 我需要 ElevatorCar（轿厢）, FloorRequest（请求队列）, State（状态枚举：静止/上行/下行）。

    • 推导结构：

      C

      struct Elevator {
    int current_floor;
    enum Direction current_dir;
    // 关键结构：一个最小堆或位图，用来存所有按下的楼层
    // 这样无论怎么断电，都知道还有哪些层没去
    BitMap pending_requests; 
};

    • 结果： 只要这个 struct 定好了，电梯的调度算法（逻辑）无论怎么改，系统都不会崩。

总结

结构直觉 = 能够看到”看不见的东西”的能力。

你看得见的是 UI 上的按钮（功能），看不见的是内存里的比特（结构）。
